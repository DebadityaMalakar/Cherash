# Cherash Interpreter Architecture

This document describes the internal design of the Cherash interpreter — how source code flows from text to execution, and how the runtime represents values, manages memory, and will support concurrency.

---

## Pipeline Overview

```
Source text (.py / .chsh)
        │
        ▼
   ┌─────────┐
   │  Lexer  │   src/lexer/mod.rs
   └─────────┘
        │  Vec<Token>
        ▼
   ┌─────────┐
   │ Parser  │   src/parser/mod.rs
   └─────────┘
        │  Module (AST)
        ▼
   ┌──────────────┐
   │ Type Checker │   src/typechecker/mod.rs   (strict mode only)
   └──────────────┘
        │  Module (validated)
        ▼
   ┌─────────────┐
   │  Evaluator  │   src/interpreter/evaluator.rs
   └─────────────┘
        │  Value / RuntimeError
        ▼
      Output
```

## Phase 1: Lexer

**File:** [src/lexer/mod.rs](../src/lexer/mod.rs)

The lexer converts raw source text into a flat `Vec<Token>`. Each `Token` carries a `TokenKind` (the variant) and a `(line, col)` source location.

### Key design decisions

**Indentation handling.** Python-style INDENT/DEDENT is generated by tracking an indent stack. When a line's leading whitespace count exceeds the top of the stack, an `INDENT` token is emitted. When it falls below, one `DEDENT` per level is emitted. Tabs expand to the next multiple of 8.

**Implicit line continuation.** A `paren_depth` counter tracks nesting of `(`, `[`, `{`. When `paren_depth > 0`, newlines are consumed silently instead of emitting `Newline` tokens.

**String prefixes.** The lexer recognises `r`, `b`, `f`, `u`, `rb`, `br`, `rf`, `fr` prefixes before quote characters, routing to different token kinds (`Str`, `Bytes`, `FString`).

**f-strings.** The content between `{}` delimiters in an f-string is extracted into `FStringPart::Expr(String)` at lex time. The expression string is re-lexed and evaluated at runtime.

## Phase 2: Parser

**File:** [src/parser/mod.rs](../src/parser/mod.rs)

The parser is a hand-written recursive descent parser with precedence climbing for expressions. It consumes `Vec<Token>` and produces a `Module` (list of `Stmt`).

### Expression precedence (low → high)

```
yield / lambda
if-else (ternary)
or
and
not
comparison  (== != < > <= >= is is not in not in)
bitwise |
bitwise ^
bitwise &
shift  (<< >>)
add / sub
mul / div / // / % / @
unary (+ - ~ await)
power (**)
postfix (call, index, attribute)
primary (literal, ident, paren, list, dict, set)
```

### AST node types

All expression variants live in `Expr`, all statement variants in `Stmt`. Both are defined in [src/parser/ast.rs](../src/parser/ast.rs).

Notable variants:
- `Expr::Compare` — carries a *chain* of ops and comparators (`a < b < c`)
- `Expr::BoolOp` — flattened n-ary `and`/`or`
- `Expr::NamedExpr` — walrus operator `:=`
- `Stmt::Try` — carries body, handlers, orelse, finalbody all in one node

## Phase 3: Evaluator

**File:** [src/interpreter/evaluator.rs](../src/interpreter/evaluator.rs)

The evaluator is a tree-walking interpreter. It walks AST nodes directly — no bytecode, no compilation step.

### Control flow via `Signal`

Functions return `EvalResult = Result<Value, Signal>`. The `Signal` enum carries:

```rust
enum Signal {
    Return(Value),   // function return
    Break,           // loop break
    Continue,        // loop continue
    Raise(RuntimeError),  // exception
}
```

This allows control flow to unwind the call stack naturally without setjmp/longjmp.

### Environment / scoping

Each scope is an `Arc<Mutex<Env>>` (aliased `EnvRef`). An `Env` holds a `HashMap<String, Value>` and an optional parent pointer.

LEGB lookup walks the parent chain. Closure capture is automatic — functions capture `env.clone()` at creation time, which is a cheap reference-count bump.

`nonlocal` is implemented by `Env::set_existing`, which walks the chain upward until it finds an existing binding and updates it in place.

### Method dispatch

When `obj.method(args)` is called:
1. `getattr_val` looks up `method` on the object.
2. If found in `cls.methods`, it is wrapped in `Value::BoundMethod { receiver, method }`.
3. `call_value` handles `BoundMethod` by prepending `receiver` to `args` before calling the underlying `Function`.

This cleanly separates attribute lookup from call semantics.

## Phase 4: Type Checker

**File:** [src/typechecker/mod.rs](../src/typechecker/mod.rs)

The type checker runs **before** the evaluator, only in strict mode. It performs a single-pass walk of the AST, inferring and checking types.

### Modes

| Mode | Activated by | Behavior |
|------|-------------|----------|
| `Lenient` | `.py` file (no header) | Type checker returns immediately with no errors |
| `Strict` | `.chsh` file or `# cherash:strict` | Full annotation checking |

### What is checked (strict mode)

- Annotated assignments: `x: int = "hello"` raises `TypeError`
- Function parameter types and return types (partial — work in progress)
- Variable binding in annotated declarations

### What is not yet checked

- Return type verification against declared `-> T`
- Generic collection element types at runtime
- Cross-function call-site argument types

These will be tightened in subsequent phases.

## Value Representation

**File:** [src/runtime/value.rs](../src/runtime/value.rs)

All Cherash values are represented as variants of:

```rust
enum Value {
    Int(i64),
    Float(f64),
    Bool(bool),
    Str(Arc<String>),
    Bytes(Arc<Vec<u8>>),
    None,
    Ellipsis,
    List(Arc<Mutex<Vec<Value>>>),
    Tuple(Arc<Vec<Value>>),
    Set(Arc<Mutex<HashSet<HashableValue>>>),
    Dict(Arc<Mutex<HashMap<HashableValue, Value>>>),
    Function(Arc<Function>),
    BoundMethod { receiver: Box<Value>, method: Arc<Function> },
    BuiltinFunction(Arc<BuiltinFn>),
    Class(Arc<Class>),
    Instance(Arc<Mutex<Instance>>),
    Module(Arc<Mutex<ModuleObj>>),
    Exception(Arc<Mutex<ExceptionObj>>),
    Range(i64, i64, i64),
}
```

### Thread safety

All mutable heap-allocated values (`List`, `Set`, `Dict`, `Instance`, `Module`) are wrapped in `Arc<Mutex<T>>`. This means:
- Values can be cheaply cloned (just a reference-count bump).
- Shared mutable state across threads is protected by the mutex.
- No GIL is required.

### Hashable values

Dict keys and set elements must be hashable. The `HashableValue` enum is a separate type that mirrors the hashable subset of `Value` (`Int`, `Float`, `Bool`, `Str`, `Bytes`, `Tuple`, `None`). Unhashable types (`List`, `Dict`, `Set`, `Instance`) raise `TypeError` when used as keys.

Float keys use a bit-cast representation (`u64`) to enable `Hash` — `NaN` is rejected at insertion.

## Standard Library

**Directory:** [src/stdlib/](../src/stdlib/)

Each module is built as a `Value::Module` containing a `HashMap<String, Value>` of its exports. Most functions are `BuiltinFn` function pointers with the signature:

```rust
fn(Vec<Value>) -> Result<Value, RuntimeError>
```

| Module | Key exports |
|--------|------------|
| `math` | `sqrt`, `sin/cos/tan`, `log`, `pi`, `e`, `factorial`, `gcd` |
| `io` | `open()`, file object with `read/write/close/__enter__/__exit__` |
| `sys` | `argv`, `exit()`, `platform`, `maxsize` |
| `threading` | `Thread`, `Lock`, `RLock` |
| `os` | `getcwd`, `listdir`, `os.path.join/exists` |
| `time` | `time()`, `sleep()` |
| `random` | `random()`, `randint()` |

## Error Handling Internals

A `RuntimeError` carries:

```rust
struct RuntimeError {
    type_name: String,     // "TypeError", "ValueError", etc.
    message: String,
    traceback: Vec<TracebackEntry>,
}
```

Errors propagate upward through the `Signal::Raise` variant of `EvalResult`. The evaluator's `Stmt::Try` handler intercepts them, matching the error's `type_name` against `except` clauses.

## Planned: Phase 7 — Real Threading

The current `threading` module runs threads synchronously (call target on `start()`). Phase 7 will replace this with:

1. `Thread::start()` spawning a real OS thread via `std::thread::spawn`
2. The spawned thread getting its own `Evaluator` instance
3. Shared `Value`s crossing thread boundaries via `Arc<Mutex<>>`
4. `Lock` backed by `std::sync::Mutex`
5. `Condition` backed by `std::sync::Condvar`

The `Value` enum is already thread-safe by construction — this is the architectural dividend of wrapping all mutable containers in `Arc<Mutex<>>` from day one.

---

## Planned: Phase 8 — GC

The current memory model is reference counting via `Arc`. This is correct but cannot collect cycles (e.g. `a.next = b; b.prev = a`).

Phase 8 will add a concurrent tricolor mark-and-sweep collector running on a background thread:

| Color | Meaning |
|-------|---------|
| White | Not yet visited. Candidate for collection at cycle end. |
| Grey | Visited; children not yet traced. |
| Black | Fully traced. All children are grey or black. |

A write barrier ensures that any pointer written while the GC is mid-cycle is correctly tracked. At cycle end, all remaining white objects are freed.

GC tuning will be exposed via environment variables:

```bash
CHERASH_GC_PERCENT      # heap growth ratio before next cycle (like Go's GOGC)
CHERASH_GC_HARD_LIMIT   # hard memory cap before forced collection
```

# Cherash Interpreter Architecture

This document describes the internal design of the Cherash interpreter — how source code flows from text to execution, and how the runtime represents values, manages memory, and supports concurrency.

---

## Pipeline Overview

```
Source text (.py / .chsh)
        │
        ▼
   ┌─────────┐
   │  Lexer  │   src/lexer/mod.rs
   └─────────┘
        │  Vec<Token>
        ▼
   ┌─────────┐
   │ Parser  │   src/parser/mod.rs
   └─────────┘
        │  Module (AST)
        ▼
   ┌──────────────┐
   │ Type Checker │   src/typechecker/mod.rs   (strict mode only)
   └──────────────┘
        │  Module (validated)
        ▼
   ┌─────────────┐
   │  Evaluator  │   src/interpreter/evaluator.rs
   └─────────────┘
        │  Value / RuntimeError
        ▼
      Output
```

## Phase 1: Lexer

**File:** `src/lexer/mod.rs`

The lexer converts raw source text into a flat `Vec<Token>`. Each `Token` carries a `TokenKind` and a `(line, col)` source location.

### Key design decisions

**Indentation.** Python-style `INDENT`/`DEDENT` is generated by tracking an indent stack. When a line's leading whitespace exceeds the top of the stack, `INDENT` is emitted. When it falls below, one `DEDENT` per level is emitted.

**Implicit line continuation.** A `paren_depth` counter tracks nesting of `(`, `[`, `{`. When `paren_depth > 0`, newlines are consumed silently.

**String prefixes.** The lexer recognises `r`, `b`, `f`, `u`, `rb`, `br`, `rf`, `fr` prefixes, routing to `Str`, `Bytes`, or `FString` token kinds.

**f-strings.** Content between `{}` delimiters is extracted into `FStringPart::Expr(String)` at lex time. The expression string is re-lexed and evaluated at runtime.

## Phase 2: Parser

**File:** `src/parser/mod.rs`

Hand-written recursive descent parser with precedence climbing for expressions. Consumes `Vec<Token>` and produces `Module { body: Vec<Stmt> }`.

### Expression precedence (low → high)

```
yield / lambda
if-else ternary
or
and
not
comparison  (== != < > <= >= is is not in not in)
bitwise |
bitwise ^
bitwise &
shift  (<< >>)
add / sub
mul / div / // / % / @
unary  (+ - ~ await)
power  (**)
postfix  (call, index, slice, attribute)
primary  (literal, ident, paren, list, dict, set, comprehension)
```

### Notable AST variants

- `Expr::Compare` — carries a chain of ops and comparators (`a < b < c`)
- `Expr::BoolOp` — flattened n-ary `and`/`or`
- `Expr::NamedExpr` — walrus operator `:=`
- `Expr::Starred` — starred expression in assignments and function calls
- `Expr::Lambda` — inline function
- `Stmt::Try` — body, handlers, orelse, finalbody in one node
- `Stmt::With` — context managers with optional `as` bindings

## Phase 3: Evaluator

**File:** `src/interpreter/evaluator.rs`

Tree-walking interpreter — walks AST nodes directly, no bytecode.

### Control flow via `Signal`

```rust
enum Signal {
    Return(Value),
    Break,
    Continue,
    Raise(RuntimeError),
}

type EvalResult = Result<Value, Signal>;
```

### Environment / scoping

Each scope is `Arc<Mutex<Env>>` (aliased `EnvRef`). An `Env` holds `HashMap<String, Value>` and an optional parent pointer.

LEGB lookup walks the parent chain. Closures capture `env.clone()` at `Function` creation time — a cheap `Arc` bump.

`nonlocal` is implemented by `Env::set_existing`, which walks the chain upward until it finds an existing binding and updates it in place.

### Keyword argument dispatch

`call_value` handles positional calls. `call_value_kwargs` handles calls with keyword arguments:

```rust
pub fn call_value_kwargs(
    &mut self,
    func: Value,
    pos_args: Vec<Value>,
    kw_args: Vec<(String, Value)>,
) -> EvalResult
```

For `Value::Function`, parameters are matched by name from `kw_args`. `ParamKind::DoubleStarArgs` collects remaining kwargs into a dict. `ParamKind::KeywordOnly` params (after `*`) must be supplied as keywords.

### Starred unpacking

Assignment targets handle `a, *b, c = seq` by finding the starred position and distributing elements before/after it:

```rust
// a, *middle, c = [1, 2, 3, 4, 5]
// → a=1, middle=[2,3,4], c=5
```

### Method dispatch

When `obj.method(args)` is called:
1. `getattr_val` looks up `method` on the object.
2. If found in `cls.methods`, it is wrapped in `Value::BoundMethod { receiver, method }`.
3. `call_value` prepends `receiver` to `args`.

For stdlib file and collection methods, methods are stored directly as `Value::BoundBuiltin` in instance fields, pre-bound at construction time.

## Phase 4: Type Checker

**File:** `src/typechecker/mod.rs`

Runs **before** the evaluator, only in strict mode. Single-pass AST walk.

### Modes

| Mode | Activated by | Behavior |
|------|-------------|----------|
| `Lenient` | `.py` file (no header) | Returns immediately with no errors |
| `Strict` | `.chsh` or `# cherash:strict` | Full annotation checking |

### What is checked

- Annotated assignments: `x: int = "hello"` → `TypeError`
- Re-assignments to annotated bindings with wrong type
- Return types against `-> T` annotations
- For-loop variable type from iterator element type
- Function and class body scoping with push/pop scope

### Type inference

The checker infers types for literals, binary ops, list/dict/set literals, calls to well-known builtins (`int()`, `str()`, `len()`, etc.), and propagates them through variable bindings.

## Value Representation

**File:** `src/runtime/value.rs`

```rust
pub enum Value {
    Int(i64),
    Float(f64),
    Bool(bool),
    Str(Arc<String>),
    Bytes(Arc<Vec<u8>>),
    None,
    Ellipsis,
    List(GcHandle<Vec<Value>>),
    Tuple(Arc<Vec<Value>>),
    Set(GcHandle<HashSet<HashableValue>>),
    Dict(GcHandle<HashMap<HashableValue, Value>>),
    Function(Arc<Function>),
    BuiltinFunction(Arc<BuiltinFn>),
    BoundMethod  { receiver: Box<Value>, method: Arc<Function> },
    BoundBuiltin { receiver: Box<Value>, name: &'static str, func: BuiltinFnPtr },
    Class(Arc<Class>),
    Instance(GcHandle<Instance>),
    Module(Arc<Mutex<ModuleObj>>),
    Exception(Arc<Mutex<ExceptionObj>>),
    Range(i64, i64, i64),
}
```

### GcHandle vs Arc<Mutex<T>>

Mutable heap values (`List`, `Dict`, `Set`, `Instance`) use `GcHandle<T>` which wraps `Arc<GcInner<T>>`:

```rust
pub struct GcInner<T> {
    pub data:  Mutex<T>,
    pub color: AtomicU8,  // GC tricolor
}
```

`GcHandle::id()` returns a stable `usize` identity used in the GC registry and threading registries. `GcHandle::clone()` is O(1) — just an `Arc` bump.

### BoundBuiltin

The `BoundBuiltin` variant solves the problem that `BuiltinFnPtr = fn(Vec<Value>) -> ...` cannot capture state. The receiver is stored separately and prepended to args at call time:

```rust
Value::BoundBuiltin { receiver, func, .. } => {
    let mut full_args = vec![*receiver];
    full_args.extend(args);
    func(full_args)
}
```

File methods in `io.rs` and collection methods use this pattern — they are stored pre-bound in instance fields.

### Hashable values

Dict keys and set elements must be hashable. `HashableValue` mirrors the hashable subset (`Int`, `Float`, `Bool`, `Str`, `Bytes`, `Tuple`, `None`). Lists, dicts, sets, and instances raise `TypeError` when used as keys. Float keys use bit-cast `u64` for `Hash`; `NaN` is rejected at insertion.

## Standard Library

**Directory:** `src/stdlib/`

Each module returns `Value::Module(Arc<Mutex<ModuleObj>>)` containing `HashMap<String, Value>` of its exports. Most functions are `BuiltinFn`:

```rust
pub type BuiltinFnPtr = fn(Vec<Value>) -> Result<Value, RuntimeError>;
```

File object methods are `BoundBuiltin` pre-bound in `Instance.fields` at `open()` time. This is necessary because `Instance.getattr` in the evaluator only auto-binds `Function` values (as `BoundMethod`), not `BuiltinFn`.

### Module registration

```rust
fn import_module(&mut self, name: &str) -> EvalResult {
    match name {
        "math"      => ok(crate::stdlib::math::make_math_module()),
        "io"        => ok(crate::stdlib::io::make_io_module()),
        "json"      => ok(crate::stdlib::json_mod::make_json_module()),
        "os"        => ok(crate::stdlib::os_mod::make_os_module()),
        "sys"       => ok(crate::stdlib::sys::make_sys_module()),
        "time"      => ok(crate::stdlib::time_mod::make_time_module()),
        "random"    => ok(crate::stdlib::random_mod::make_random_module()),
        "threading" => ok(crate::stdlib::threading_mod::make_threading_module()),
        _ => err(RuntimeError::new("ImportError", ...)),
    }
}
```

## Error Handling Internals

`RuntimeError` carries:

```rust
pub struct RuntimeError {
    pub type_name: String,
    pub message:   String,
    pub traceback: Vec<TracebackEntry>,
    pub cause:     Option<Arc<Mutex<ExceptionObj>>>,
    pub context:   Option<Arc<Mutex<ExceptionObj>>>,
}
```

Errors propagate via `Signal::Raise`. `Stmt::Try` intercepts them, matching `type_name` against `except` clauses. Exception chaining (`raise X from Y`) sets `.cause`. Bare `raise` re-raises the current exception via `Signal::Raise`.

## GC Integration

The GC background thread is started once by `init_gc()` at program startup. All heap objects register themselves at allocation via `alloc_list`, `alloc_dict`, `alloc_set`, `alloc_instance`. The GC registry holds type-erased `GcRecord` closures for color access, child tracing, and clearing.

See [GC.MD](GC.MD) for full design documentation.

---

## Threading Integration

Each `Thread.start()` spawns a real OS thread with its own `Evaluator` instance. Threading primitives (`Lock`, `Condition`, etc.) store their Rust-side state in global registries keyed by instance GC ID, since non-Clone types cannot live inside `Value`.

See [THREADING.MD](THREADING.MD) for full design documentation.
